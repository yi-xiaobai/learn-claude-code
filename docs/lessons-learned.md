# 踩坑记录

> 从 87 个 commit 里总结的教训。大白话，别再犯。

## 1. 拆太细，最后还得合

turtle 的构建、发布、构建发布，拆成了三个 skill，写了一堆重复代码，最后还是合成一个。

**下次**：干的是同一件事就别拆开，用子命令搞定。

## 2. 密码先提交了再想起来忽略

.env 里的密码、token 先提交了，后面才补 .gitignore，来回补了好几次。

**下次**：第一个文件写 .gitignore，密码类的东西永远别进 git。

## 3. 什么都往里塞，后面一直在删

半成品功能、截图、team-plugins、测试文件……全往仓库里丢，后面花了 10 多个 commit 清理。

**下次**：想清楚这个仓库放什么。没做完的东西开分支，别往 master 上扔。

## 4. 代码写了，文档忘了

功能提交了，README 没更新。过几天想起来再补一个 docs commit，反反复复补了七八次。

**下次**：改代码和改文档放在同一个 commit 里，一起提交。

## 5. 每个 skill 结构都不一样

早期每个 skill 想怎么写就怎么写，后来统一结构的时候挨个重构，白费功夫。

**下次**：先定模板（SKILL.md + scripts/），所有 skill 照着来。

## 6. 写完就提交，用的时候才发现有 bug

好几个 fix 都是自己跑的时候才发现的问题：忘了 git pull、权限报错、弹窗没处理。

**下次**：提交前自己跑一遍完整流程，别偷懒。

---

## 新项目第一天该做的

1. 写 `.gitignore` — 先把坑堵上
2. 写 `CLAUDE.md` — 定规矩
3. 定 skill 模板 — 统一结构
4. 配 pre-commit — 自动检查
5. README 跟代码一起更新 — 别拆开
6. 告诉 LLM 做什么 无需太细致 — 写好 CLAUDE.md
