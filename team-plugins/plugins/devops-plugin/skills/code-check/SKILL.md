# Code Check - 提交前自检

在代码提交前进行自我检查，只分析 git 改动的文件（staged + unstaged）。

## 检查范围

1. 获取当前仓库的 git 改动文件：
   ```bash
   git diff --name-only HEAD
   git diff --name-only --cached
   ```
2. 排除 node_modules、dist、build、.git 等目录

## 检查维度

### 1. 语法与编译
- 使用项目已有的 linter（ESLint）检查
- 运行 `npm run lint` 或项目配置的 lint 命令

### 2. 潜在 bug
- 空指针/空数组访问
- 异步/await 错误处理遗漏
- 条件判断逻辑错误（== vs ===，赋值 vs 比较）
- 副作用未处理

### 3. 代码风格
- 变量/函数命名（是否清晰、无拼写错误）
- 重复代码（可提取为函数/常量）
- 注释是否过期或缺失

### 4. 安全问题
- 硬编码的密钥、密码、Token
- SQL 注入风险（拼接字符串作为 SQL）
- XSS 风险（innerHTML、dangerouslySetInnerHTML）
- 敏感信息 console.log

### 5. 性能问题
- 不必要的重复计算
- 大数据未分页
- 循环内异步调用

## 输出格式

按严重程度分类输出：

```
🚨 Error（必须修复）
  - [文件名] 行号: 问题描述

⚠️ Warning（建议修复）
  - [文件名] 行号: 问题描述

💡 Suggestion（可选优化）
  - [文件名] 行号: 问题描述
```

如果无问题，输出：
```
✅ 代码检查通过，无明显问题
```

## 检查流程

1. 获取 git 改动的文件列表
2. 对每个文件进行静态分析
3. 运行项目 linter（如果有）
4. 汇总问题并输出报告
5. 给出总体评价

## 注意事项

- 只检查改动的文件，不要检查整个项目
- 如果是新增文件，同样进行检查
- 如果项目没有配置 lint，提示用户可以添加 ESLint
- 保持报告简洁，聚焦关键问题
